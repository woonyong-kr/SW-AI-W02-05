# 자료구조 (Data Structure)

## 자료구조란?

자료구조는 컴퓨터 메모리 공간에 데이터를 효율적으로 배치하고 조직화하는 방법이다. 쉽게 말하면 **'정리 기술'**이다.

- 데이터가 '물건'이라면, 자료구조는 그 물건들을 담을 **선반, 상자, 파일첩**과 같다.
- 단순히 저장하는 것에서 끝나지 않고, **읽기(Search), 추가(Insert), 삭제(Delete), 수정(Update)**를 얼마나 빠르고 편하게 할 수 있는지가 핵심이다.

---

## 왜 필요한가?

컴퓨터의 자원(CPU, 메모리)은 한정되어 있고 비용과 직결된다. 데이터가 수백만, 수천만 개로 늘어나면 어떤 구조를 선택했느냐에 따라 성능 차이가 크게 벌어진다.

- **시간 효율성 (Time Complexity)**: 원하는 데이터를 얼마나 빨리 찾고 조작할 수 있는가
- **공간 효율성 (Space Complexity)**: 메모리를 얼마나 적게 쓰면서 데이터를 저장할 수 있는가

> 100만 권의 책이 무작위로 쌓인 창고 vs 가나다순으로 정리된 창고에서 특정 책을 찾는 속도 차이를 생각해보면 직관적으로 이해된다.

---

## 자료구조의 종류

| 분류 | 대표 자료구조 |
|------|-------------|
| 순서가 중요한 문제 | 배열(Array), 리스트(List), 스택(Stack), 큐(Queue), 데크(Deque), 연결 리스트(Linked List) |
| 관계가 중요한 문제 | 트리(Tree), 이진 탐색 트리(BST), 자가균형 트리(AVL / Red-Black), B트리(B-Tree), 힙(Heap), 트라이(Trie) |
| 연결이 중요한 문제 | 그래프(Graph) |
| 속도가 중요한 문제 | 해시 테이블(Hash Table), 딕셔너리(Dictionary), 맵(Map), 셋(Set) |

---

## 순서가 중요한 문제

### 배열 (Array)

"비슷한 물건들을 나란히 **연속된 사물함**에 보관하는 것"이다.

- 데이터를 메모리 공간에 연속해서 저장하기 때문에, 시작 주소만 알면 몇 번째 데이터든 즉시 접근할 수 있다.
- 크기가 고정적이고, 같은 타입의 데이터만 담을 수 있다.

**C언어 배열 vs 파이썬 리스트 - 내부 동작 차이**

언어마다 메모리를 다루는 방식이 다르다.

| 구분 | C언어 배열 (원시 배열) | 파이썬 리스트 (동적 배열 + 참조) |
|------|----------------------|-------------------------------|
| 사물함에 저장되는 것 | 실제 데이터(값) | 데이터의 주소(포인터) |
| 타입 제한 | 같은 타입만 가능 | 혼합 가능 |
| 크기 | 선언 시 고정 | 동적으로 확장 가능 |

- C언어: 정수 3개짜리 배열이라면 실제 메모리에도 정수 3개가 빈틈없이 연속으로 붙어 있다.
- 파이썬: 리스트 자체의 메모리 공간에는 값들이 어디 있는지 가리키는 주소(화살표)들만 나란히 모여 있다. 덕분에 숫자, 문자, 객체 등 크기와 종류가 다른 데이터를 섞어서 담을 수 있다.

---

### 스택 (Stack)

"접시를 쌓아 올리는 구조"다.

- 데이터를 위에서만 넣고, 위에서만 꺼낼 수 있다. **LIFO(Last In First Out)**, 나중에 들어온 것이 먼저 나간다.
- 주요 연산: `push`(삽입), `pop`(꺼내기), `peek`(맨 위 확인)
- 활용 예시: 함수 호출 스택, 브라우저 뒤로 가기, 괄호 유효성 검사

---

### 큐 (Queue)

"줄을 서는 구조"다.

- 먼저 들어온 데이터가 먼저 나간다. **FIFO(First In First Out)**.
- 주요 연산: `enqueue`(삽입), `dequeue`(꺼내기)
- 데이터를 넣은 순서대로 처리해야 하는 상황이 생기면서 고안되었다.
- 활용 예시: 프린터 대기열, 은행 번호표, 서버 요청 처리

---

### 데크 (Deque, Double-Ended Queue)

"양쪽 끝에서 모두 넣고 꺼낼 수 있는 구조"다.

- 큐(앞에서만 꺼냄)와 스택(뒤에서만 넣고 꺼냄)의 특성을 합쳐놓은 형태다. **앞(front)과 뒤(rear) 양방향**으로 삽입·삭제가 모두 가능하다.
- Python에서는 `collections.deque`로 제공되며, 양쪽 끝 연산이 모두 **O(1)**이다. 리스트의 `pop(0)`은 O(n)이지만 deque의 `popleft()`는 O(1)이라 큐를 구현할 때 리스트보다 deque가 훨씬 효율적이다.
- 주요 연산: `append`(오른쪽 추가), `appendleft`(왼쪽 추가), `pop`(오른쪽 제거), `popleft`(왼쪽 제거)
- 활용 예시: 슬라이딩 윈도우 알고리즘, 브라우저 앞/뒤로 가기 기록, 작업 스케줄링

---

### 연결 리스트 (Linked List)

"각 사물함이 다음 사물함의 주소를 함께 적어두는 구조"다.

- 데이터를 메모리 어딘가에 분산해서 저장하고, 각 노드가 다음 노드의 주소를 가리키는 방식으로 연결한다.
- 배열과 달리 크기가 유동적이고, 중간 삽입·삭제가 빠르다. 대신 특정 위치를 찾으려면 처음부터 순서대로 따라가야 한다.
- 활용 예시: 음악 플레이어 재생 목록, 실행 취소(Undo) 기능

---

## 관계가 중요한 문제

### 트리 (Tree)

"뿌리(Root)에서 가지가 뻗어나가는 계층 구조"다.

- 하나의 루트 노드에서 시작해 자식 노드들이 연결되며 뻗어나간다. 사이클(순환)이 없다.
- 가족관계도나 컴퓨터의 폴더 구조처럼 계층적인 데이터를 다루기 위해 만들어졌다.
- 주요 개념: 루트(Root), 부모/자식 노드, 리프(Leaf) 노드, 깊이(Depth), 높이(Height)
- 활용 예시: 파일 시스템, 조직도, HTML DOM

---

### 이진 탐색 트리 (Binary Search Tree, BST)

"왼쪽은 작은 값, 오른쪽은 큰 값"이라는 규칙이 있는 트리다.

- 모든 노드에서 왼쪽 자식은 자신보다 작고, 오른쪽 자식은 자신보다 크다.
- 이 규칙 덕분에 이진 탐색처럼 절반씩 범위를 줄여가며 데이터를 찾을 수 있다.
- 균형 잡힌 트리라면 탐색 시간복잡도는 **O(log n)**이다.
- 단점: 데이터가 정렬된 순서로 삽입되면 한쪽으로 치우쳐 선형 구조(최악 O(n))가 된다.

---

### 자가균형 트리 (Self-Balancing Tree)

"스스로 균형을 유지하는 이진 탐색 트리"다.

BST의 단점인 편향 문제를 해결하기 위해 고안되었다. 삽입·삭제 시 자동으로 트리의 높이를 균형 있게 조정한다.

**AVL 트리**
- 모든 노드에서 왼쪽과 오른쪽 서브트리의 높이 차이가 최대 1을 유지한다.
- 균형 조건이 엄격해 탐색 속도가 빠르지만, 삽입·삭제 시 회전(Rotation) 연산이 자주 발생한다.

**레드-블랙 트리 (Red-Black Tree)**
- 각 노드에 빨강/검정 색상을 부여하고 특정 규칙을 통해 균형을 유지한다.
- AVL보다 균형 조건이 느슨해 삽입·삭제가 빠르다. 실무에서 가장 많이 사용되는 균형 트리다.
- 활용 예시: Java의 `TreeMap`, C++ STL의 `map`, Linux 커널 스케줄러

---

### B트리 (B-Tree)

"한 노드가 여러 개의 키와 자식을 가질 수 있는 트리"다.

- 이진 탐색 트리가 자식을 최대 2개 가지는 것과 달리, B트리는 하나의 노드에 여러 키를 저장하고 여러 자식을 가진다.
- 디스크 접근 횟수를 최소화하도록 설계되어 있어, 메모리가 아닌 디스크 기반 저장소에 적합하다.
- 트리의 높이가 낮아지기 때문에 탐색에 필요한 I/O 횟수가 줄어든다.
- **B+트리**: B트리의 변형으로, 실제 데이터는 리프 노드에만 저장하고 리프 노드끼리 연결 리스트로 이어진다. 범위 탐색에 유리하다.
- 활용 예시: 관계형 데이터베이스(MySQL InnoDB) 인덱스, 파일 시스템

---

### 힙 (Heap)

"항상 가장 크거나 가장 작은 값이 맨 위에 있는 트리"다.

- 완전 이진 트리 구조를 유지하면서, 부모 노드가 자식 노드보다 항상 크거나(최대 힙) 작다(최소 힙).
- 최댓값 또는 최솟값을 O(1)로 꺼낼 수 있다.
- 활용 예시: 우선순위 큐, 다익스트라 알고리즘, 힙 정렬

---

## 연결이 중요한 문제

### 그래프 (Graph)

"노드(정점)와 노드를 잇는 간선(Edge)으로 이루어진 구조"다.

- 트리와 달리 사이클이 있을 수 있고, 방향이 있을 수도(방향 그래프), 없을 수도(무방향 그래프) 있다.
- 지도상의 경로 탐색이나 친구 추천처럼 복잡한 관계망을 표현하기 위해 탄생했다.
- 주요 탐색 방법: BFS(너비 우선 탐색), DFS(깊이 우선 탐색)
- 활용 예시: 네비게이션 경로 탐색, SNS 친구 추천, 네트워크 라우팅

---

## 속도가 중요한 문제

### 해시 테이블 (Hash Table)

"이름만 대면 바로 위치를 알 수 있는 구조"다.

- 키(Key)를 해시 함수(Hash Function)에 넣으면 저장 위치(인덱스)가 계산된다. 탐색·삽입·삭제 모두 평균 **O(1)**이다.
- 서로 다른 키가 같은 위치로 계산되는 **해시 충돌(Collision)**이 발생할 수 있으며, 이를 해결하는 방법이 성능을 좌우한다.
- 활용 예시: 로그인 세션 관리, 캐싱, 데이터베이스 인덱싱

---

### 딕셔너리 / 맵 / 셋 (Dictionary / Map / Set)

해시 테이블을 기반으로 각 언어에서 제공하는 고수준 자료구조다.

| 이름 | 설명 | 대표 언어 |
|------|------|----------|
| 딕셔너리(Dictionary) | 키-값 쌍으로 데이터를 저장 | Python `dict` |
| 맵(Map) | 딕셔너리와 동일한 개념, 순서 보장 버전도 존재 | Java `HashMap`, JavaScript `Map` |
| 셋(Set) | 중복 없는 값의 집합, 값의 존재 여부 확인에 특화 | Python `set`, Java `HashSet` |

- 세 가지 모두 내부적으로 해시 테이블을 사용하기 때문에 평균 O(1)의 빠른 접근이 가능하다.
- 활용 예시: 중복 제거(Set), 빠른 키 기반 조회(Map/Dict), 카운팅(Dict)
